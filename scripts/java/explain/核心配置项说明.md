# 核心配置项说明

本文只说明源码里已经出现、并且会影响运行行为的关键配置项。

## 1. 文档主题概述

这一份文档负责汇总项目里真正会影响运行行为的配置开关、端口、前缀、自动配置入口和环境差异。

它不是“配置清单抄录”，而是把配置和具体功能关联起来，方便维护时直接回到源码链路。

## 4. 关键实现说明

### 4.1 安全、租户和 Redis 是联动的

修改 `security.excludes`、`tenant.excludes` 或 Redis 前缀时，不能只看单独一个配置项，要一起检查调用链和缓存键。

### 4.2 一些开关是自动配置入口

像 `captcha.enable`、`api-decrypt.enabled`、`mybatis-encryptor.enable`、`sse.enabled`、`websocket.enabled`、`warm-flow.enabled` 这种开关，会直接影响相关自动配置是否生效。

### 4.3 环境差异主要集中在外部依赖

开发和生产环境最主要的差异通常是：

- 数据源地址
- Redis 地址
- 监控客户端
- SnailJob 客户端
- 上传目录

## 5. 使用方式或示例

### 5.1 典型配置写法

```yaml
sa-token:
  token-name: Authorization

tenant:
  enable: false

sse:
  enabled: true
  path: /resource/sse
```

### 5.2 修改配置时的顺序

1. 先确认功能在哪个模块里实现
2. 再确认对应自动配置类或控制器
3. 最后改 `application.yml` 或 profile 文件

这样不容易只改配置却没有同步代码约束。

## 6. 注意事项

- `spring.profiles.active` 由构建链路注入，不能只看单个 yml
- `spring.boot.admin.client.*` 和 `snail-job.*` 在示例环境里不一定都启用
- 示例密码和地址不能直接拿去生产使用
- `tenant.enable` 关闭时，租户相关逻辑不会按开启状态运行

## 7. 待确认项

- `spring.profiles.active` 的最终取值链路是否完全由 Maven profile 注入，仍需结合打包命令和父级 `pom.xml` 再核实
- `spring.boot.admin.client.*` 与 `snail-job.*` 的生产接入参数是否全部与真实部署一致，需要以环境变量或发布配置为准
- `warm-flow.token-name` 的多 token 语义是否在所有调用场景下都按当前配置生效，建议和前端鉴权链路一起确认
